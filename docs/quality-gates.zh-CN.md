# 质量门禁

本项目因平台发布质量而存在。功能不能仅因为平台 API 接受了请求就算完成。渲染的审查体验必须可读、稳定且可安全重跑。

## 完成定义

每个实现任务在报告完成前必须满足以下相关门禁。

所有任务必需：

- 变更行为的单元测试
- `go test ./... -count=1` 通过
- `go vet ./...` 通过
- `go build ./...` 通过
- `git diff --check` 通过
- 已跟踪文件中无令牌、本地主机名或本地文件系统路径

解析器变更必需：

- 使用真实 OCR 输出的固定测试
- JSON 对象前有非 JSON 文本的测试
- 未知未来字段的测试
- 缺失可选字段的测试
- 对至少一个真实 OCR 版本捕获的输出固定的兼容性测试

渲染器变更必需：

- 黄金 Markdown 测试
- 已有代码在围栏中，不得为裸文本
- 已知文件扩展名使用语言感知围栏
- 未知文件扩展名回退到 `text`
- 诊断在 `<details>` 内渲染
- 渲染的评论中不出现原始 GitLab/GitHub API 错误文本
- 不引入平台特定的建议围栏（除非有专门设计）

GitLab 发布变更必需：

- 请求方法、路径、查询和正文的 `httptest` 覆盖
- GitLab 13.12 回退行为已测试
- 使用 `X-Next-Page` 测试讨论分页
- 对差异固定测试内联锚点选择
- clear/update 行为仅删除或更新标记的发布器拥有的评论

公开发布前必需：

- PR CI 已通过
- 定时/最新 OCR 兼容性检查已通过，或发布前记录了已知不兼容性
- 可选的 GitLab e2e 测试对真实 GitLab 实例通过
- 本地冒烟脚本将真实 OCR 输出发布到测试 MR 并获取评论进行断言
- 重复发布更新摘要而不创建重复项
- clear 操作不触及未标记的用户评论
- 文档包含限制和支持的平台/版本范围
- 英文和中文 README 文件都存在且对齐
- README 徽章遵循本地 readme-badges 技能

## 渲染评论检查清单

从真实 MR 获取的评论必须满足：

- 产品头部或徽章可见
- 发现文本清晰，无需阅读原始 OCR JSON
- 位置信息存在（可用时）
- 建议变更可读且不误导
- 已有代码在围栏代码块中
- 审查上下文在冗长时折叠
- 摘要显示总发现数和内联跳过计数
- 发布诊断已折叠
- 稳定标记存在但不显眼
- 重复发布后无重复摘要

## 发布阻止器

以下任何情况都会阻止发布：

- 评论包含原始平台错误，如 `line_code can't be blank`
- 已有代码渲染为一行裸散文
- 重复发布创建重复摘要评论
- clear 删除了未标记的用户或第三方机器人评论
- GitLab 13.12 e2e 无法运行或无法在记录环境需求后明确跳过
- CLI 打印令牌或密钥
- JSON 模式向 stdout 输出人类聊天内容
- 最新支持的 OCR 输出无法再解析
- 兼容性工作流无法检测 OCR 输出契约变更

## OCR 兼容性门禁

OCR 是上游依赖，尽管它作为 CLI 执行。其输出契约必须用真实二进制测试。

必需的兼容性覆盖：

- 最低支持 OCR 版本的已签入固定
- 发布时验证的最新 OCR 版本的已签入固定
- 接受两个固定的解析器测试
- 可从指定 OCR 版本重新生成固定的本地脚本
- 安装最新已发布 OCR 包并验证解析器兼容性的定时 GitHub Action

兼容性作业不得发布评论或需要 GitLab 令牌。如果实时 OCR 生成需要 LLM 凭据，工作流必须分离：

- 固定兼容性，始终运行
- 最新 OCR 实时生成，仅在配置了凭据时运行

定时 CI 发现的任何输出格式变更应产生带有捕获的脱敏输出摘录的问题或失败工作流。

## 审查实践

每次工作交接应包含：

- 精确的任务范围
- 必需的测试和冒烟检查
- 提醒渲染的评论必须通过获取的平台笔记检查，而非仅本地 Markdown 字符串
- 要求命令输出、变更文件和跳过的门禁的最终报告

审查交接应包含：

- 变更文件的链接
- 精确的渲染 Markdown 示例或黄金固定名称
- 任何真实 GitLab MR 冒烟结果
- 已知限制和延迟行为
